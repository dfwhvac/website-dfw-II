## Step 1 — Hero image (biggest single lever)

### Audit first
```bash
# Run Lighthouse on the homepage
npx lighthouse https://dfwhvac.com --view --preset=desktop
npx lighthouse https://dfwhvac.com --view --form-factor=mobile --throttling-method=simulate
```

Look at the **"Largest Contentful Paint element"** in the report. It will name the exact DOM node + asset URL.

### Likely findings
The LCP element is almost certainly one of:
- The hero image on the homepage (`/`)
- The h1 + headline text (rare on text-only pages)

If it's an image, check:
- **Format**: Is it JPEG or PNG? Should be AVIF (or WebP fallback).
- **Size**: Is the file >150KB? Should be <80KB on mobile.
- **Dimensions**: Is a 1920×1080 hero being served to a 390px mobile viewport? Should serve responsive sizes via `sizes` attribute.
- **Loading**: Is `<Image priority />` set? Is there a `<link rel="preload">` in the document head?

### Implementation
```jsx
// In components/HomePage.jsx hero section
import Image from 'next/image'

<Image
  src="/hero-mobile.avif"        // Or use Sanity's image pipeline
  alt="DFW HVAC technician"
  priority                       // ← critical: tells Next.js to NOT lazy-load
  width={1920}
  height={1080}
  sizes="(max-width: 768px) 100vw, 100vw"
  className="..."
/>
```

And in `app/layout.js` `<head>`:
```jsx
<link
  rel="preload"
  as="image"
  href="/hero-mobile.avif"
  fetchpriority="high"
  media="(max-width: 768px)"
/>
```

If the hero comes from Sanity:
- Use `urlFor(image).format('avif').width(800).quality(75).url()` for mobile
- Use `urlFor(image).format('avif').width(1920).quality(80).url()` for desktop
- Serve via Sanity's image CDN (already free, already fast, already supports AVIF)

### Verify
After deploy, re-run Lighthouse. LCP should drop 400-700ms.

## Step 2 — Font preload + drop unused weights

### Audit
```bash
grep -rn "from 'next/font'" /app/frontend/app /app/frontend/components
grep -rn "import.*Inter\|Roboto\|Open Sans" /app/frontend
```

### Check
- Are all font weights you import (400, 500, 600, 700, 900?) actually used above the fold?
- Most sites only need 2 weights visible in the hero. Drop the rest.
- Is `preload: true` set in the `next/font` config?
- Is `display: 'swap'` set? (Should be — prevents FOIT.)

### Implementation
```js
// app/layout.js
import { Inter } from 'next/font/google'

const inter = Inter({
  subsets: ['latin'],
  weight: ['400', '700'],     // ← only what's needed above the fold
  display: 'swap',            // ← prevents invisible text during font load
  preload: true,              // ← critical
  variable: '--font-inter',
})
```

Then verify in the rendered HTML that there's a `<link rel="preload" as="font" type="font/woff2">` tag.

## Step 3 — Defer render-blocking JS

### Audit
```bash
grep -rn "Script\|<script" /app/frontend/components /app/frontend/app
```

### Check each `<Script>` tag for its `strategy` prop:

| Script | Current strategy | Should be |
|---|---|---|
| Microsoft Clarity | check | `lazyOnload` |
| Google Analytics 4 | check | `afterInteractive` (needed for SPA route tracking) |
| reCAPTCHA v3 | check | `lazyOnload` (load on first form interaction, not on page load) |
| RealWork widget | check | `lazyOnload` (already lazy via IntersectionObserver — verify the script tag itself) |
| Vercel Analytics | check | auto-handled by `@vercel/analytics` package |

### Implementation
```jsx
import Script from 'next/script'

<Script
  src="https://www.clarity.ms/tag/XXXXXX"
  strategy="lazyOnload"
/>
```

**Critical**: ensure no script uses `strategy="beforeInteractive"` — that one is render-blocking by design.

### Watch out for
- reCAPTCHA needs to load before the form renders. If you `lazyOnload` it, the form's first interaction will have a 1-2s delay while the script downloads. Mitigation: load reCAPTCHA when the user first focuses any form field (use `onFocus` to dynamically inject the script tag).

## Step 4 — TTFB / Sanity CDN / cache tuning

### Verify Sanity client config
```bash
grep -n "api.sanity.io\|apicdn.sanity.io" /app/frontend/lib/sanity.js
```

The Sanity client should use the CDN endpoint:
```js
// lib/sanity.js
export const client = createClient({
  projectId: 'iar2b790',
  dataset: 'production',
  apiVersion: '2024-01-01',
  useCdn: true,                // ← critical — uses apicdn.sanity.io
})
```

`useCdn: true` is the difference between a 50ms response (CDN) and a 300ms response (origin API).

### Add `cache: 'force-cache'` to critical GROQ fetches
```js
// In fetch helpers used by homepage hero / above-the-fold data
const data = await client.fetch(query, params, { cache: 'force-cache' })
```

This tells Next.js to use its built-in fetch cache aggressively. Combined with ISR `revalidate = 3600` (still active until P2.18 changes it), means most requests hit Vercel's edge cache and never round-trip to Sanity at all.

### Verify Vercel function region
1. Vercel Dashboard → Project → Settings → Functions → **Region**
2. Should be `iad1` (US East / Washington DC) or `cle1` (US East / Cleveland) for Dallas audience. Both are <50ms.
3. If it's set to `cdg1` (Paris) or `hnd1` (Tokyo), that's adding 100-200ms TTFB unnecessarily.

## Validation protocol

After each step:

1. **Deploy to Vercel preview branch.**
2. **Run PageSpeed Insights** on the preview URL (mobile + desktop): https://pagespeed.web.dev/
3. **Compare to baseline.** Did LCP drop by at least 80ms?
4. If **yes** → merge to main, move to next step.
5. If **no** → the fix didn't address the actual bottleneck. STOP. Don't ship. Investigate why the predicted gain didn't materialize before continuing.

The "don't ship a step that didn't move the needle ≥80ms" rule is a guardrail against assumption-based optimization. The whole reason we're sequencing this with measurement gates is that LCP is multi-factor — sometimes a fix that looks great on paper does nothing because another bottleneck is hiding it.
