# P2.18 — Next.js 16 `cacheComponents` Spike Findings

**Status:** ⏸ DEFERRED — non-trivial migration required
**Spike date:** Feb 2026
**Time spent:** ~30 min (build attempt + decision)
**Outcome:** Discard branch, defer to dedicated session

---

## What was tested

1. Added `cacheComponents: true` at the top level of `next.config.js` (Next 16's promoted-out-of-experimental flag — replaces `experimental.ppr` and `experimental.dynamicIO`).
2. Ran `yarn build` against the current codebase.

## What the build reported

Hard-fail on **every page route** with the error:

> `Route segment config "revalidate" is not compatible with nextConfig.cacheComponents. Please remove it.`

The flag forces the new explicit cache model:
- Pages become **dynamic by default**.
- ISR is no longer expressed via `export const revalidate = N`.
- Cached behavior must be opted in per fetch / per component using the `'use cache'` directive plus `cacheLife()` / `cacheTag()` from `next/cache`.

## Decision

**Defer**, with the spike's evidence baked into the ROADMAP.

### Reasoning
- The hypothesis behind P2.18 was a **30-min, low-risk LCP win**.
- Empirical result: 24-route refactor — every Sanity fetch in `lib/sanity.js` needs a `'use cache'` wrapper, every page must drop `revalidate`, and the hero subtree needs explicit `<Suspense>` boundaries to actually benefit.
- The risk/reward shifts from "spike" to "major refactor": potential breakage of LocalBusiness schema, GBP review badge counts, GA4 server-side events, Sanity ISR semantics, and the `/api/canonical-description` ETag flow.
- Tailwind v4 (Sequence 4) is a higher-confidence ROI session in the same effort window.

## Conditions to revisit

Reopen as a full migration (a dedicated 4-6 hr session) **only when**:
1. Sequence 4 (Tailwind v4) has shipped — keeps blast radius isolated.
2. The Microsoft Clarity baseline (ends May 27, 2026) has completed and given us before/after LCP data to compare against.
3. We have a clear LCP regression in CrUX field data on the homepage hero (`/`, `/services/*`, `/cities-served/*`) above ~50ms p75.

Until then, ISR (`revalidate = 3600`) + Vercel edge cache continues to serve the use case adequately. No degradation observed in current CWV reports.

## Migration sketch (for the future session)

If/when re-attempted, the migration order is:

1. Wrap all Sanity GROQ helpers in `lib/sanity.js` with `'use cache'` + `cacheTag('sanity:<doc-type>')` + `cacheLife('hours')`.
2. Remove all 24 `export const revalidate = 3600` lines.
3. Wrap each page's data-fetch promise tree in `<Suspense fallback={...}>`.
4. Add `revalidateTag('sanity:companyInfo')` etc. to the existing `/api/revalidate` webhook for Sanity content updates.
5. Verify GA4 + reCAPTCHA + LeadForm submit flow still server-renders correctly (forms are client components — should be unaffected).
6. Confirm `LocalBusinessSchema` JSON-LD still streams in the document head (must not regress AEO).
7. Baseline LCP via Lighthouse + PSI before, then after.
